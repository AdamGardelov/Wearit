# Wearit Image Assets and Storage Lifecycle Design

**Date:** 2026-07-18  
**Status:** Approved in conversation

## Goal

Make Wearit's locally prepared garment bundles deterministic, storage-efficient,
and safe to replace. The image preparation skill and the application must share
one explicit contract for coordinate-aligned wear layers, browsable product
images, generated outfit thumbnails, and obsolete Storage objects.

## Asset contract

### Canonical mannequin

- Opaque 8-bit sRGB PNG.
- Exactly `887x1774` pixels.
- Provides the coordinate plane used by every wear layer.

### Wear layer

- Lossless, metadata-free, optimized 8-bit sRGB RGBA PNG.
- Exactly `887x1774` pixels, with both transparent and visible pixels.
- Contains the garment at its final mannequin coordinates; it is not a tightly
  cropped garment image.
- Uses the same coordinate plane as the mannequin. Its neutral transform is:

  ```json
  {
    "anchorX": 0.5,
    "anchorY": 0.5,
    "scale": 1,
    "rotationDegrees": 0
  }
  ```

- Small reviewed corrections may adjust the transform, but category-based
  cropped-garment defaults such as scales from `0.3` to `0.75` must not be
  applied to canonical full-canvas layers.
- Dual-chroma removal contracts the matte enough to remove dark generated
  fringe, feathers the boundary, and despills only edge pixels. It must preserve
  the accepted fit and the full canvas.

### Product image

- The bundle builder, rather than the caller, creates the final WebP derivative.
- 8-bit sRGB WebP, encoder quality `88`, with metadata removed.
- Maximum longest edge `1600` pixels, with aspect ratio preserved and no
  enlargement of smaller sources.
- Transparent cutout with both fully transparent and visible pixels. Opaque
  product sources are rejected instead of receiving a baked background.
- Alpha is preserved through WebP encoding so the garment floats directly on
  Wearit's current or future page background without a rectangular image field.
- Exactly one front image is primary. At most one back image is allowed;
  additional images use the detail view.
- The application displays product images with `object-fit: contain`; their
  aspect ratio is not fixed by the storage contract.

Encoder quality is an input to derivative creation and is not treated as a
property that can be recovered reliably by inspecting an existing WebP file.

### Outfit thumbnail

- Opaque WebP generated by the application.
- Exactly `600x1200` pixels.
- Encoder quality `0.86`.

### Processing-only images

Raw photographs, opaque masters, chroma sources, fit masters, review previews,
and mannequin previews remain local and are not included in the bundle or
uploaded to Supabase.

## Bundle preparation

The v2 bundle builder owns final derivative creation:

1. Validate the source manifest, stable item/image UUIDs, image roles, and
   canonical wear-layer dimensions.
2. Require transparent product cutouts, auto-orient them, resize within
   `1600x1600` without enlargement, remove metadata, and encode WebP at quality
   `88` with alpha quality `100`.
3. Strip metadata and losslessly optimize each canonical wear-layer PNG without
   changing its pixels, dimensions, alpha semantics, or coordinate alignment.
4. Emit product assets under stable `.webp` bundle names and the wear layer as
   `assets/<item-id>/wear-layer.png`.
5. Report product-image bytes, wear-layer bytes, and total bundle bytes in both
   build and dry-run output.
6. Preserve deterministic output: rebuilding unchanged reviewed inputs must end
   with a dry-run that reports no changes.

The Admin bundle reader and database contract continue to require PNG for wear
layers and accept the builder's WebP product assets.

## Supabase paths and replacement lifecycle

New imports use immutable versioned paths:

```text
<owner-id>/items/<item-id>/wear-layer/<version>.png
<owner-id>/items/<item-id>/images/<image-id>-<version>.webp
<owner-id>/outfits/<outfit-id>/thumbnail-<version>.webp
```

For a wardrobe item replacement, the repository reads the currently referenced
wear-layer and structured product-image paths before uploading. The lifecycle is:

1. Upload every new versioned object.
2. Atomically update the database references through the existing import RPC.
3. Only after the RPC commits, best-effort delete prior referenced objects that
   differ from the new paths.
4. Never roll back the committed database row because obsolete-object deletion
   failed. Return a cleanup warning instead.
5. If upload or the RPC fails before commit, remove only the newly uploaded
   objects, preserving all prior referenced objects.

The existing owner-scoped orphan reconciliation remains the fallback for failed
post-commit cleanup and historical orphans. It revalidates candidates against
current database references immediately before deletion.

## Error handling

- Reject non-canonical wear-layer dimensions and invalid alpha before creating a
  bundle.
- Reject malformed image roles, duplicate stable IDs, and missing sources.
- A derivative encoding failure leaves the previous bundle untouched.
- A partial upload or failed import RPC cleans up the newly uploaded paths.
- A failed obsolete-object deletion produces a recoverable warning containing
  no destructive retry against current database paths.

## Verification

- Builder tests prove transparent PNG input becomes transparent WebP, opaque
  product sources are rejected, large sources are reduced without enlargement,
  output names are `.webp`, and wear layers remain canonical optimized RGBA
  PNGs.
- Builder tests prove build summaries contain bytes by asset class and total.
- Repository tests prove prior paths are deleted only after a successful RPC,
  are preserved on RPC failure, and cleanup failure returns a warning while the
  import remains committed.
- Skill regression tests prove a two-pixel dark chroma fringe becomes a smooth,
  consistent edge without moving the accepted garment.
- The seven pants items are regenerated from their reviewed chroma sources,
  deterministically previewed on the canonical mannequin, visually reviewed,
  bundled, and followed by an unchanged dry-run.
